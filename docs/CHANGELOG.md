### Up-Front Pick'em → survivor pool (no knockout cascade) (2026-06-08)

- **Why.** In Pick'em the knockout was a matchup bracket seeded from the player's own predicted groups, so one wrong group call cascaded through every downstream matchup. Pick'em is now a **survivor pool**: you pick *which teams advance* each round, independent of who they play, so a missed group pick only costs that team's points. **Bracket Reset is unchanged.**
- **Selection.** From your 32 qualifiers (12 groups × 1st/2nd + 8 third-place) you choose the **16** that reach the Round of 16, then **8** → QF, **4** → SF, **2** → Final, **1** champion — each round a subset of the last. Stored in the same `r32/r16/qf/sf/final` columns as **sets** (no schema change). `lib/bracket.ts` adds `pool32`, `survivorCandidates`, `survivorPicks`, `SURVIVOR_COUNTS`, `toggleSurvivor`, `prunePickemSurvivors`, and `pickGroupPickem` / `pickThirdPickem` (these re-validate the survivor sets instead of wiping the knockout, so swapping a group's 1st/2nd — same two teams — keeps every pick).
- **Scoring (`lib/bracket-scoring.ts`).** `scoreBracketEntry(entry, results, mode)` is now **mode-aware**: `pickem` scores each knockout round by **set membership** (the picked team is anywhere in that round's actual advancer set, dedup'd, partial-result-safe), `reset` keeps **bracket position**. Same point values and the same **231** max per mode; breakdown labels shift one round ahead for pickem ("Round of 16", "Quarter-finals", …, "Champion"). New `knockoutPickCorrect(nm, slot, actuals, isPickem)` drives the ✓/✗ chips (pickem stays *pending* until a round is fully decided, so a team mid-run isn't shown wrong). `score-utils.ts` passes `entry.mode`.
- **UI.** New **survivor picker** on the Bracket tab for Pick'em (narrow 32 → 16 → 8 → 4 → 2 → champion as tappable team chips); Reset keeps the matchup tree. Groups / 3rd-place tabs use the pruning handlers in pickem. The Summary tab and the league member view (`BracketReviewer` / `BracketDetail`) read correctness from the mode-aware breakdown + `knockoutPickCorrect`, with pickem-shifted round labels. `BracketEntry` gains a `mode` field.
- **Verification.** Logic re-run standalone (sandbox mount can't compile the edited tree — see note below): 22/22 assertions — cap enforcement, subset cascade on deselect, *no* cascade on a group 1st/2nd swap, third-place pruning, order-independent + dedup'd + partial-safe set scoring, `knockoutPickCorrect` pending/decided, perfect pickem = 231. Added pickem cases to `bracket-scoring.test.ts`. Run `npm test` / `npm run build` on the host to confirm types end-to-end.

### Fix: cron deploy failing on Cloudflare's 5-trigger account cap (2026-06-08)

- **Symptom.** CI deploy of `soccer-fantasy-cron` failed at the schedule-registration step with `code: 10072` — *"You have exceeded the limit of 5 cron triggers."* The Worker uploaded fine; only the cron registration was rejected. Cloudflare counts cron triggers **per account** (cap 5 on the Free plan), across every Worker — and the leftover pre-rebrand `wcfantasy-cron` (3 triggers) plus this Worker's 3 pushed the account over 5.
- **Fix — single-trigger dispatch.** `cron-worker/wrangler.toml` now registers **one** trigger (`*/5 * * * *`) instead of three. `cron-worker/src/index.ts` adds a pure, exported `stepsForTick(scheduledTimeMs)` that selects the job groups from the tick's UTC time: every-5-min jobs always run; the 6-hourly group runs when `minute == 0 && hour % 6 == 0` (00/06/12/18); the daily reminder when `hour == 18 && minute == 0`. This works because `*/5` already fires at every `:00`. Account-wide cron usage for this Worker drops from **3 → 1**. The `scheduled()` handler now dispatches off `controller.scheduledTime` rather than `controller.cron`.
- **Verified.** `tsc --strict` clean; simulated all 288 daily ticks — base jobs on every tick, 6-hourly group exactly 4×/day, daily reminder exactly once at 18:00, correct ordering (at 18:00 all three groups run in sequence), zero discrepancies.
- **Still recommended.** Delete the abandoned `wcfantasy` / `wcfantasy-cron` Workers to reclaim their 3 trigger slots (already flagged as cleanup in `DEPLOY-CLOUDFLARE.md`). The single-trigger change makes the deploy resilient, but only frees budget if at least one of the 5 slots is open.

### Picks scoring moved to a set-based DB RPC; Stats/global standings fix (2026-06-07)

Fixes the **"Score picks → scored N picks across 0 users"** bug: after a group-stage sim the Stats → **Picks** tab (and global standings) showed **0 points**, even though each pick's `points_earned` and the league standings were correct.

- **Root cause.** `lib/score-utils.ts#scorePicks` fanned out one Supabase REST call **per pick** (`Promise.all` over ~140 updates) plus per-user / per-league upserts. On Cloudflare Workers that exceeds the subrequest / 6-concurrent-connection limit — the same limit already worked around for match writes via `bulk_finish_matches` — so writes failed nondeterministically: `picks.scored_at` sometimes never persisted (the aggregation read then saw **0 users**), and when it did, `global_scores` was frequently left un-aggregated while `league_scores` updated, so the Stats page (which reads `global_scores`) showed 0.
- **Fix — `public.score_picks()` RPC.** Does the whole pipeline in **one DB round-trip**: scores all unscored picks for FINISHED matches, rebuilds per-user aggregates, upserts `global_scores` (preserving `bracket_points`) and `league_scores`, and recomputes ranks. Idempotent — re-running repairs a zeroed/partial aggregation. `scorePicks()` now just calls the RPC; the simulate route's auto-score path inherits the fix. Migration `add_score_picks_rpc` (in-repo: `migrations/2026-06-07_score_picks_rpc.sql`).
- **Reset fix.** The Sim **Reset** zeroed `global_scores` but omitted `exact_scores`, leaving a stale value behind (e.g. `exact_scores=6` with everything else 0). Now zeroed too (`src/app/api/admin/simulate/route.ts`).
- **Production backfill.** Ran `score_picks()` against prod to repair existing rows — `global_scores.picks_points` 0 → 72 for both members, global ranks restored (totals now match `league_scores`: 88 / 86).

### Live automation — cron, live-score sync, knockout fixtures, realtime wiring (2026-06-06)

Implements the gap report's **Phase 2 ("make it live")** critical path.

- **Cron worker rewritten** (`cron-worker/`). Previously only `*/15 → score-bracket` was mapped and the declared `*/10` trigger had no handler, so live-score sync, pick scoring, and reminders never ran on a schedule. Now each schedule runs an ordered step list:
  - `*/5 * * * *` → `sync-scores` then `score-picks` (pull live/finished scores, then score newly-finished picks)
  - `0 */6 * * *` → `sync-fixtures` then `score-bracket` (pick up knockout fixtures as they resolve, rescore brackets)
  - `0 18 * * *` → `notify-picks-reminder` (daily reminder email)
  - Calls are uniform `POST`s; a startup check warns if `APP_URL` is missing the `/soccer-fantasy` basePath (the previous silent-404 footgun).
- **`sync-scores` now captures LIVE matches.** `mapStatus` mapped in-play games to `SCHEDULED`, so the `/live` page was always empty; it now maps `IN_PLAY`/`PAUSED → LIVE` (and fetches the full competition, updating only in-play/finished/postponed rows).
- **New `/api/sync-fixtures`** — idempotent full-schedule sync that generates **knockout match rows (R32 → Final)** as the draw/results resolve them. Unlike the one-shot `bootstrap-matches`, it does **not** create team rows (the cause of the duplicate-team bug); it maps football-data team ids onto existing teams via `api_id` and skips fixtures whose teams are still TBD. Wired to the 6-hour cron and exposed as an admin **"Sync fixtures"** button.
- **Shared auth `lib/api-auth.ts` (`isCronOrAdmin`)** applied to `sync-scores`, `sync-fixtures`, `score-picks`, `score-bracket`, `seed-teams`, `notify-picks-reminder`. All now accept **both GET and POST**. This fixes the admin **"Sync live scores"** and **"Seed teams"** buttons, which previously returned 401 (they called CRON-secret-only routes with no header).
